★ Core principle

Discover first. Build second. Every prompt Claude writes for Cursor starts with the same instruction: query the live org before writing a single line of code. This eliminates the #1 source of build failures — field name assumptions from training data that don’t match the actual data model.

The process
The prompt cadence
1
Adam → Claude

Describe the demo moment

Adam explains the demo story in business terms — not the technical spec. What the customer will see, what problem it solves, what impression it should leave.

2
Claude

Write the Cursor prompt

Claude translates the narrative into a precise Cursor prompt — always starting with "query the org first." No field name assumptions. No guesses from training data.

3
Cursor + MCP

Query the org → Build

Cursor hits the live Salesforce org via MCP, confirms real field API names via DESCRIBE, then generates the full LWC bundle — JS, HTML, XML, Apex — using actual field names.

4
Claude → Deploy

Review → Deploy

Claude audits output against the demo story. Does the logic match? Are Record Type IDs dynamic? Apex first, then LWC. Always --target-org. Deploy success ≠ logic works — always verify on a real record.
